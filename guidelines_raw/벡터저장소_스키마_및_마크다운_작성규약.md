# 벡터 저장소 스키마와 마크다운 작성 규약

감사조서–감사기준 연결 파이프라인의 2단계(문단 저장소) 설계 문서다. 0단계에서 원문을 마크다운으로 변환할 때 4장의 규약을 지키면, 파서가 사람 손 없이 2장의 스키마를 전부 채울 수 있도록 두 부분이 서로 맞물려 설계되어 있다. 개정 이력은 문서 끝(6장)에 둔다.

---

## 1. 설계 원칙 — 스키마는 "받아야 할 질문"이 결정한다

이 저장소가 감당해야 할 질의는 일곱 가지이며, 모든 필드는 이 일곱 질의 중 어느 것을 위해 존재하는지가 분명해야 한다.

| 번호 | 질의 유형 | 예시 |
|---|---|---|
| ① | 정확 라우팅 | 조서 2100 해설 시 기준서 315의 전 문단을 그대로 인출 |
| ② | 범위 내 의미 검색 | 점검 항목 한 문장을 기준서 315 문단들에 정밀 대응 |
| ③ | 전체 범위 의미 검색 | 지정 기준서에서 답이 부족할 때만 저장소 전체로 확장 |
| ④ | 인용 대조 | 해설이 인용한 "315 문단 25"가 실재하는지 식별자로 즉시 조회 |
| ⑤ | 문맥 확장 | 찾은 문단의 앞뒤 문단을 붙여서 가져오기 |
| ⑥ | 문단 성격 필터 | 요구사항 문단을 적용지침 문단보다 우선 배치 |
| ⑦ | 상호참조 확장 | 검색에 걸린 발췌 대조표가 가리키는 원전 문단을 끌어오기 (3.7) |

## 2. 레코드 구조

### 2.1 레코드 한 건의 실제 모습

회계기준서 제1115호 문단 31(수행의무 이행 시 수익 인식 — 조서 4000P-1의 5단계 모형 중 5단계와 직결)을 예로 든 저장 단위 한 건이다.

```json
{
  "id": "KIFRS::1115::31",
  "document": "고객에게 약속한 재화나 용역, 즉 자산을 이전하여 수행의무를 이행할 때(또는 기간에 걸쳐 이행하는 대로) 수익을 인식한다. …(이하 원문 그대로)",
  "embedding": [0.021, -0.117, …],
  "metadata": {
    "source_type":    "회계기준",
    "standard_no":    "1115",
    "standard_title": "고객과의 계약에서 생기는 수익",
    "para_no":        "31",
    "para_type":      "본문",
    "section_path":   "수행의무의 이행",
    "seq":            74,
    "has_table":      false,
    "char_len":       118,
    "source_file":    "kifrs_1115.md"
  }
}
```

핵심 설계 포인트가 하나 있다. `document`에 저장하는 글(화면 표시와 인용에 쓰는 깨끗한 원문)과 `embedding`을 계산할 때 넣는 글(검색용 합성문)이 서로 다르다는 점이다. 이유와 방법은 3.2절에서 설명한다.

### 2.2 필드 정의표

| 필드 | 타입 | 예시 | 존재 이유 (대응 질의) |
|---|---|---|---|
| `id` | 문자열 | `KSA::315::A124` | ④ 인용 대조기의 직접 조회 열쇠. 문법은 `유형코드::기준서번호::문단번호[#순번]` — `#순번`은 분할 레코드에만 붙는다(3.6) |
| `source_type` | 문자열 | 감사기준 / 회계기준 / 실무지침 | 세 원문 묶음의 구분 필터 |
| `standard_no` | 문자열 | `"315"` / `"1115"` / `"2016-1"` | ①② 정확 라우팅 필터의 핵심. 실무지침 번호("2016-1") 때문에 반드시 문자열 |
| `standard_title` | 문자열 | 고객과의 계약에서 생기는 수익 | 임베딩 머리말 재료, 리포트 표기 |
| `para_no` | 문자열 | `31` / `A124` / `한138.1` | 인용 표기의 원천. 접두 문자(A, 한)를 그대로 보존. **분할 레코드에서도 분할 전 논리 문단 번호를 유지한다**(3.6) |
| `para_type` | 문자열 | 요구사항 / 적용지침 / 본문 / 부록 / 참조 / 정의 | ⑥ 성격 필터. 문단번호 접두와 절 제목에서 자동 유도 — 유도 규칙은 표 아래 |
| `section_path` | 문자열 | 수행의무의 이행 | 임베딩 문맥 재료, 리포트 소제목 |
| `seq` | 정수 | 74 | ⑤ 문맥 확장 — 같은 기준서 안에서 순번 ±1~2 조회 |
| `has_table` | 불리언 | false | 표를 통째로 담은 예외 문단의 추적 |
| `char_len` | 정수 | 118 | 길이 이상치 점검(누락·과병합 탐지) |
| `source_file` | 문자열 | kifrs_1115.md | 원본 파일 역추적 |
| `part_no` | 정수 (선택) | 2 | 분할 레코드 한정(3.6). 부재 = 통짜 문단 |
| `part_total` | 정수 (선택) | 3 | 분할 정합성 검증의 기준값(점검 7) |
| `cross_refs` | 문자열 (선택) | `KIFRS::CF::2.11,KSA::320::4` | 상호참조 간선 — 참조를 주장하는 쪽이 보유(3.7). 채움은 v2 백로그 |

식별자의 유형코드는 세 가지로 고정한다: `KSA`(감사기준), `KIFRS`(회계기준), `GUIDE`(실무지침).

**para_type 유도 규칙(확정판).** 감사기준 — 숫자 = 요구사항, A접두 = 적용지침, `부록-N`·`보론N-M` = 적용지침(원문 번호를 보유한 정본 부속). 회계기준 — 숫자·한접두 = 본문, 알파벳 접두(A/B/C/D/E/BA 등) = 적용지침(정본 부속). 실무지침 — 숫자 = 본문. 공통 — `부록N`·`부록-사례N`·`사례X` = **부록**(예시류: 검색 기본 후순위·옵트인 대상), `참조N` = **참조**(검색 기본 포함), `정의-{용어}` = **정의**(부록 A 용어정의 표의 행 단위 조각 — 검색 기본 포함이며 질의 ⑥에서 정의 우선 배치의 근거이자, 용어 사전 파생 색인의 셀렉터).

### 2.3 구현상 주의점 세 가지

1. **메타데이터 타입 제약.** 벡터 데이터베이스(Chroma 기준)의 메타데이터에는 문자열·정수·실수·불리언만 들어가고 목록(list)은 넣을 수 없다. 다중값이 필요한 필드는 `"KSA::330::6,KSA::240::26"`처럼 쉼표로 이어 붙인 문자열로 넣는다. 이 형식의 정식 사례가 상호참조 간선 `cross_refs`이며, 소유와 탐색 규칙은 3.7에 따른다.
2. **구분자.** 하이픈은 "2016-1" 같은 기준서 번호와 충돌하므로, 번호에 절대 등장하지 않는 `::`를 식별자 구분자로 쓴다. 같은 원칙으로 분할 순번 구분자는 `#`를 쓴다(3.6).
3. **라우팅 표는 저장소 밖에 둔다.** 문단 메타데이터에 "이 문단을 참조하는 조서 목록"을 역방향으로 심으면, 조서 서식이 바뀔 때마다 수만 개 문단을 전부 갱신해야 하는 결합이 생긴다. 조서→기준서 라우팅 표는 별도의 작은 조회표(제이슨 파일 또는 SQLite)로 유지한다.

## 3. 설계 갈림길과 결정 사항

### 3.1 컬렉션 구성 — 단일 컬렉션에 `source_type` 필터

코퍼스별 3개 컬렉션은 개정 시 통째 재구축이 쉽다는 장점이 있으나, 이 프로젝트에서는 한 조서가 두 코퍼스에 걸치는 경우가 흔하다(8600 감사의견 형성은 감사기준 700·705·570·450, 조서 4000P-1은 회계기준 1115). 나눠 두면 조서 하나 해설에 여러 번 질의하고 결과를 병합해야 하므로, **단일 컬렉션 + `source_type` 필터**로 간다.

판본 관리는 **컬렉션 이름이 단독으로 담당한다.** 컬렉션 이름에 기준선 시점을 박아 두고(예: `standards_20250829` — 조서 서식들이 명시한 유효시점), 기준서가 개정되면 새 이름의 컬렉션을 새로 만든다. 리포트에는 어느 컬렉션으로 생성했는지를 기록해, 인용 대조가 항상 같은 판본을 보게 한다.

### 3.2 임베딩 입력의 합성 — 품질을 가장 크게 좌우하는 결정

기준서 문단은 "감사인은 ~을 문서화하여야 한다"처럼 짧고 규범적인 문장이 많아, 본문만 임베딩하면 서로 다른 기준서의 비슷한 문장들이 의미 공간에서 뒤엉킨다. 저장은 원문 그대로 하되, 임베딩 계산에는 아래처럼 머리말을 합성해 넣는다.

```
회계기준 1115 고객과의 계약에서 생기는 수익 | 수행의무의 이행 | (문단 본문)
```

용어집 조각(`정의-{용어}`, 4.3의 네 번째 갈래)도 같은 틀을 따른다 — 본문이 `{용어}: (정의문)` 형태이므로 머리말 합성만으로 정의 질의(②)에 맞는 입력이 된다.

```
회계기준 1115 고객과의 계약에서 생기는 수익 | 부록 A 용어의 정의 | 수행의무: (정의문)
```

화면 표시와 인용은 깨끗한 원문(`document`)을, 검색은 문맥이 밴 합성문을 쓰는 이중 구조다. 임베딩은 저장소 바깥에서 직접 계산해 넣는다. 모델은 한국어 성능이 검증된 공개 변환 모델(bge-m3 계열, 출력 1024차원), 거리 척도는 코사인을 권장한다.

### 3.3 식별자에는 판본을 넣지 않는다

`KIFRS::1115::31`이 판본과 무관하게 안정되어 있어야 인용 대조기와 라우팅 표가 개정 때마다 깨지지 않는다. 판본 구분은 3.1의 컬렉션 이름이 담당한다.

### 3.4 단어 일치 색인은 바깥에 두고, 식별자만 공유한다

감사 용어는 "핵심감사사항", "왜곡표시"처럼 글자 그대로의 일치가 승부처인 경우가 많아 의미 검색만으로는 부족하다. 단어 일치 색인(한국어 형태소 분석기 kiwipiepy + BM25 방식)은 벡터 저장소 바깥에 별도로 만들고, 두 색인이 **동일한 `id` 체계를 공유**하게 해서 검색 결과를 순위 합산(역순위 융합)으로 병합한다. 스키마에 요구되는 약속은 "같은 식별자를 쓴다" 단 하나다.

### 3.5 규모 감각

감사기준 전체 + 회계기준서 40여 종 + 실무지침을 합쳐도 문단 수는 대략 1~2만 개 수준으로, 어떤 저장소를 쓰든 일반 노트북에서 가볍게 돌아간다. 이 문서는 Chroma 기준으로 적었으나 Qdrant, pgvector로 옮겨도 그대로 이식된다.

### 3.6 대형 문단의 분할 표시 — id는 주소, 메타데이터는 사실

임베딩 입력 한도(bge-m3 기준 8,192토큰) 등으로 한 논리 문단을 여러 조각으로 나눠야 할 때 적용한다. 현재 대상은 `KSA::1200::부록1`(9,036자)과 `KSA::240::부록1`(6,024자) 2건이다.

```json
{
  "id": "KSA::1200::부록1#2",
  "metadata": {
    "standard_no": "1200",
    "para_no":     "부록1",
    "part_no":     2,
    "part_total":  3
  }
}
```
(다른 필드는 통상과 동일하므로 생략)

1. **id에는 유일성에 필요한 최소한만 넣는다.** 저장소가 레코드마다 유일한 id를 요구하므로 한 문단이 세 레코드가 되는 순간 구별자가 필요하다. 무작위가 아닌 결정적 `#순번`이어야 재적재 업서트와 재현성이 유지되며(id 원칙과 동일), 구분자 `#`는 `::`와 같은 이유로 — 문단번호에 절대 등장하지 않는 문자여서 — 선택했다.
2. **para_no는 분할 전 논리 문단 번호를 그대로 유지한다.** 인용의 단위는 문단이지 조각이 아니다. 생성물은 "기준서 1200 부록1"을 인용하지 "부록1#2"를 인용하지 않으므로, 인용 계층은 분할의 존재를 몰라도 된다. 인용 대조기는 2단계로 동작한다: 정확 id 조회 → 실패 시 메타데이터 등가 조회(`standard_no` + `para_no`).
3. **"분할됐다"는 사실은 메타데이터(part_no, part_total)가 담는다.** "부록1의 모든 조각"류 질의를 id 접두 문자열 수술로 풀면 저장소 이식성이 깨진다 — 메타데이터 등가 비교는 모든 저장소가 지원하지만 id 접두 매칭은 저장소마다 다르다. 메타데이터로는 조건 한 줄(`standard_no`=…, `para_no`=…)이다.
4. **part 필드는 분할 레코드에만 넣는다.** 부재 = 통짜 문단으로 해석한다. 예외 2건 때문에 일반 레코드 9,500여 개에 필드 세금을 물리지 않는다.
5. **임베딩 머리말에 조각 표기를 더한다:** `… | 부록1 (2/3) | (조각 본문)` — 검색기가 부분 조각임을 알게 한다. 분할선은 표·목록 경계 등 의미 경계를 우선한다.

요약: **id는 불변의 주소, 메타데이터는 질의 가능한 사실.** 판본을 id가 아닌 컬렉션 이름에 맡긴 3.3과 같은 계열의 판단이다.

### 3.7 상호참조의 소유와 탐색 — 간선은 참조를 주장하는 쪽에

실무서 2의 부록(개념체계·기준서 발췌 대조표, `참조N` 조각·para_type="참조")처럼 문서 간 상호참조가 존재할 때, "어느 방향으로 이어줄 것인가"라는 질문은 **저장 방향**과 **탐색 방향**으로 쪼개면 풀린다.

1. **저장 — 참조를 주장하는 쪽이 간선을 갖는다.** 대조표는 실무서 2 저자들이 편찬한 실무서 2의 자산이므로 `cross_refs`는 실무서 2 레코드에 둔다(형식: 2.2의 쉼표 연결 id 문자열). 반대로 원전(개념체계) 문단에 "나를 참조함"을 심으면 실무서 2가 개정될 때마다 원전 레코드를 갱신하는 역결합이 생긴다 — 라우팅 표를 저장소 밖에 두기로 한 2.3-3과 동일한 논리다. 이 원칙은 감사기준 문단의 내부 참조("(문단 A5, A7 및 A10 참조)")에도 그대로 일반화된다: 참조하는 문단이 간선을 갖는다.
2. **역방향 — 저장하지 않고 파생한다.** 적재 시 `cross_refs`를 한 번 훑어 "대상 id → 참조자 id 목록" 역색인을 저장소 밖의 작은 조회표로 생성한다. 단어 일치 색인(3.4)과 같은 패턴 — 원본은 한 곳, 파생물은 언제든 재생성 가능.
3. **탐색 — 양방향을 열되 비대칭으로 연다.**
   - 지침→원전: **자동 확장.** 참조 조각이 검색에 걸리면 그것이 가리키는 원전 문단을 끌어와 문맥에 포함한다. 생성 가드("발췌 대조표 인용 시 원전 문단을 우선 인용")가 작동하려면 원전이 문맥에 실제로 있어야 하기 때문이다.
   - 원전→지침: 기본은 **포인터만** 단다("이 문단의 적용 지침: 실무서 2 문단 7, 17"). 본문 자동 첨부는 문맥 낭비인 데다 실무서의 비강제 지위와도 맞지 않고, 질의 의도가 적용·문서화이거나 라우팅 옵트인일 때만 본문으로 승격한다 — 예시류 삼중 통제와 같은 모양새다.
4. **단계화 — v1은 이미 동작한다.** `참조N` 텍스트 조각 자체가 매핑을 산문으로 담고 있어 검색에 걸리면 모델이 그대로 읽는다. 구조화된 `cross_refs` 채움 + 역색인 + 자동 확장은 v2 백로그로 두고, 집합 질의("중요성 관련 규정 전부")의 실제 빈도를 질의 로그에서 확인한 뒤 투자한다.

## 4. 마크다운 작성 규약 (0단계)

원문 변환이 아래 규약만 지키면, 파서가 사람 손 없이 2장의 스키마를 전부 채울 수 있다.

### 4.1 파일 예시

```markdown
---
source_type: 회계기준
standard_no: "1115"
standard_title: 고객과의 계약에서 생기는 수익
---

## 수행의무의 이행

31. 고객에게 약속한 재화나 용역(자산)을 이전하여 수행의무를
이행할 때(또는 기간에 걸쳐 이행하는 대로) 수익을 인식한다. …

32. 다음 …

## 기간에 걸쳐 이행하는 수행의무

35. 다음 기준 중 어느 하나를 충족하면 …
```

### 4.2 규칙 여섯 가지

1. **문서 정보 구획.** 파일 첫머리의 문서 정보 구획(front matter)에 스키마 필드명을 그대로 쓴다. 필수 필드는 `source_type`, `standard_no`, `standard_title` 세 가지이며, `standard_no`는 따옴표로 감싸 문자열임을 고정한다.
2. **절 제목.** 마크다운 제목 기호 `##`로만 표기한다. 이것이 그 절에 속한 문단들의 `section_path`가 된다.
3. **문단 절단선.** 문단은 반드시 행 머리에 원문 번호 그대로 `31.` `A124.` `한138.1.` 형태로 시작하고, 다음 번호가 나오기 전까지가 한 문단이다. 행 머리의 `번호.` 패턴 하나로 기계 절단이 가능해진다.
4. **표.** 표는 그 표가 속한 문단 번호 아래에 마크다운 표로 붙인다. 파서가 이를 해당 문단에 합쳐 한 조각으로 담고 `has_table: true`를 기록한다. 표를 중간에서 자르면 의미가 부서지므로 통째 유지가 원칙이다.
5. **각주.** 본문 사이에 `[각주: …]` 형태로 끌어올려 넣는다. 각주가 문서 하단에 흩어져 있으면 문단 절단이 깨지는 것이 이 작업의 단골 사고다.
6. **제거 대상.** 목차, 머리글·바닥글, 쪽번호는 제거한다.

### 4.3 예외 규칙 — 번호 없는 덩어리 (확정판)

원문에 문단 번호가 없는 덩어리는 성격에 따라 네 갈래로 처리한다.

1. **예시류**(예시 감사보고서, 예시 목록): 사례 경계 — "사례 N" 헤더행, 사례별 절 제목이 있으면 절 경계 — 로 분할한다. ID는 `부록N`(도입·목록부), `부록-사례N`(감사기준 사례), `사례X`(실무서 2의 사례 A~T처럼 원문 라벨을 보유한 경우). para_type="부록"으로 유도되어 검색 기본 후순위·옵트인 대상이 된다.
2. **참조·발췌 대조표**(예: 실무서 2 부록 "개념체계와 기준서 참조"): 원천 제목(굵은 글씨) 경계로 분할한다. ID는 `참조N`, para_type="참조"(검색 기본 포함). 소유·탐색 규칙은 3.7.
3. **용어집**(부록 A "용어의 정의" 표 — 신기준 계열 16개와 감사기준 1200의 보론 1): 표의 행 = 용어 하나로 분할한다. ID는 `정의-{정규화된 용어명}`(용어명은 원문 표기에서 공백만 제거, 괄호·가운뎃점·하이픈 보존), para_type="정의"(검색 기본 포함), 본문은 `{용어}: (정의문)`. **4.2의 표 통짜 원칙(규칙 4)의 명시적 예외**다 — 용어집 표는 행이 각각 독립 의미 단위라, 행·열이 상호의존하는 일반 표와 성격이 다르기 때문이다. 부록 A 안의 "다음 용어는 타 기준서에서 정의한다" 안내문 + 용어 나열 블록은 둘째 갈래(`참조N`)로 처리한다.
4. **그 밖의 덩어리**: 절 제목 단위 한 조각으로 두고 ID는 `부록N`(초판 규칙 유지).

어느 갈래든 조각당 6,000자 미만을 지향하고, 초과가 불가피한 진성 장문은 3.6의 분할 표시를 적용한다.

### 4.4 파일 단위와 이름

기준서(또는 실무지침) 한 건 = 마크다운 파일 하나로 한다. 파일명은 `유형_번호.md` 꼴로 통일하며(`ksa_315.md`, `kifrs_1115.md`, `guide_2016-1.md`), 이 이름이 그대로 `source_file` 필드에 들어간다.

## 5. 적재 후 점검 목록 — 2단계 완료 기준

1. 기준서별 문단 수를 원문 목차와 표본 대조한다.
2. `standard_no`별로 `seq`의 연속성을 검사해 구멍(누락 문단)을 찾는다.
3. `id` 중복이 0건인지 확인한다.
4. `has_table` 문단 몇 건을 눈으로 열어 표가 부서지지 않았는지 확인한다.
5. 임베딩 개수와 레코드 수가 일치하는지 확인한다.
6. 무마침표 문단머리 의심 행(행 머리가 `A1 본문`처럼 마침표 없이 시작하는 행) 스캔이 0건인지 확인한다
   — 원본의 부록 구역에서 문단 절단이 새는 것을 잡는 검사다.
7. **분할 정합성.** `part_total`이 있는 레코드에 대해, 같은 (`standard_no`, `para_no`) 조합의 조각이 part_no 1부터 part_total까지 결번·중복 없이 존재하는지 확인한다.
8. **상호참조 실재성(v2).** `cross_refs`를 채우는 시점부터, 나열된 모든 대상 id가 같은 컬렉션에 실재하는지 대조한다.

1~7 항목이 모두 통과되어야 3단계(두 갈래 검색 조립)로 넘어갈 자격이 생긴다. 8번은 `cross_refs`를 구조화하는 v2부터 적용한다.

- **1차 개정** (코퍼스 결함 수리, 커밋 ea03347 반영분). 점검 6(무마침표 문단머리 스캔) 추가 — 회계기준 부록 구역의 문단 절단 누락 160건을 잡아낸 검사를 상설화.
- **2차 개정**. ① 질의 ⑦(상호참조 확장) 추가. ② id 문법 `[#순번]` 확장과 분할 표시 원칙 신설(3.6): id는 주소, 메타데이터는 사실, para_no는 논리 번호 유지. ③ 상호참조 소유·탐색 원칙 신설(3.7): 간선은 참조하는 쪽에, 역방향은 파생 색인, 탐색은 비대칭 옵트인. ④ 선택 필드 `part_no`·`part_total`·`cross_refs` 추가. ⑤ para_type "참조" 신설 및 유도 규칙 확정판 명문화. ⑥ 규약 4.3 확정판(예시류·참조·기타 세 갈래). ⑦ 점검 7(분할 정합성)·8(상호참조 실재성, v2) 추가. ⑧ 2.1 예시의 문단 31 자구를 코퍼스 검증 결과대로 교정("재화나 용역, 즉 자산을").
- **3차 개정** (본판, v2.1 — 수정 8 반영). ① 규약 4.3에 네 번째 갈래 **용어집** 추가: 부록 A 용어정의 표를 행(용어) 단위로 분할, ID `정의-{용어}`. 표 통짜 원칙(4.2 규칙 4)의 예외 근거 명시 — 용어집 표는 행=독립 의미 단위. ② para_type **"정의"** 신설 및 유도 규칙에 추가(`정의-` 접두 → 정의, 검색 기본 포함). ③ 3.2에 용어집 조각의 임베딩 머리말 예시 추가. ④ 용어명 정규화 규칙(공백만 제거, 괄호·가운뎃점·하이픈 보존, 각주 마커 제거)은 후속 용어 사전 구축기가 동일하게 사용한다.
